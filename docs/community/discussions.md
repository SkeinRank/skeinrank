# GitHub Discussions

SkeinRank uses GitHub Discussions for questions, ideas, architecture tradeoffs, integration feedback, and release conversations.

Use Issues for reproducible bugs and concrete implementation tasks. Use Discussions when the topic is exploratory, architectural, or not yet ready to become an actionable issue.

## Categories

| Category | Use it for |
| --- | --- |
| Announcements | Maintainer updates, releases, roadmap notes, and project status. |
| Q&A | Setup help, SDK/CLI usage, runtime API questions, bindings, snapshots, MCP, and integration questions. |
| Ideas | Product suggestions, workflow improvements, and early feature ideas. |
| Architecture / RFC | Design tradeoffs for runtime routing, governance, delivery, MCP, RAG, and search integrations. |
| Integrations | Elasticsearch/OpenSearch, MCP clients, RAG frameworks, search backends, GitOps, and operator-controlled delivery. |
| Show and tell | Demos, experiments, screenshots, integrations, and usage notes from real projects. |

## Issues vs Discussions

| Use Issues for | Use Discussions for |
| --- | --- |
| Reproducible bugs | Setup questions |
| Failing commands, tests, or CI | Architecture tradeoffs |
| Documentation mistakes | Product feedback |
| Concrete feature requests with expected behavior | Early ideas and RFCs |
| Regression reports | Integration exploration |

## Maintainer workflow

1. Keep `Announcements` for maintainer-owned posts.
2. Move questions and RFCs out of Issues when they do not yet describe an actionable change.
3. Link accepted Discussions back to implementation Issues or PRs.
4. Keep category names, issue forms, and labels aligned with [`github-labels.md`](github-labels.md).
5. Keep release-specific announcements concise and link to the GitHub Release notes for full details.
